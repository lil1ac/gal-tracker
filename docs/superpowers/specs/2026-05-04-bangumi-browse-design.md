# Bangumi 游戏浏览页重构设计

## 目标

将现有“浏览全部游戏”页面重构为稳定的 Bangumi 游戏发现工作台。用户应能直接查看游戏条目、按常用维度筛选、翻页浏览、查看详情，并把条目加入本地库。

## API 策略

使用 Bangumi v0 API：

- `POST /v0/search/subjects` 用于列表浏览，固定 `filter.type = [4]` 限定游戏。
- `GET /v0/subjects/{subject_id}` 用于详情面板。
- 默认浏览使用 `keyword: "galgame"` 和热门排序，避免空关键词在搜索接口中产生不稳定结果。
- 保留 `match`、`heat`、`rank`、`score` 排序，以及年份、评分、排名、标签、NSFW 过滤。

## 交互结构

Browse 页面由三块组成：

- 顶部筛选栏：关键词、分类预设、排序、年份、评分、排名、标签、NSFW、搜索/重置。
- 中间结果区：响应式封面网格，显示封面、标题、年份、评分、排名、入库状态。
- 详情面板：显示封面、简介、评分、排名、收藏统计、标签、平台、Bangumi 链接和“添加到我的库”按钮。

## 错误与状态

- 初次加载显示骨架屏。
- 翻页和筛选时保留旧结果并显示顶部进度条。
- 搜索失败时展示可重试错误，不清空已有结果。
- 详情加载失败时在页面级错误区域提示。
- 添加入库时做重复保护，成功后本地状态立即更新为“已入库”。

## 代码边界

- `bangumiMeta.ts` 负责 API 请求体构建、请求、响应归一化。
- Browse 组件只处理页面状态和交互。
- 新增转换工具，集中把 Bangumi 条目或详情转换成本地 `Game`，避免组件间重复拼对象。

## 验证

- 增加/更新服务层测试，覆盖默认游戏搜索、空关键词兜底、分页参数和 Bangumi 到本地游戏转换。
- 运行 `npm run build`。
- 运行 `npm test`，若既有无关测试仍失败，需要明确记录失败位置。
